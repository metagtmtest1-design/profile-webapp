# Portfolio Website — Vertical Implementation Plan (GitHub + Cloudflare)

## Understanding Confirmation
Single-page portfolio (Hero, Services 6-grid, About, Testimonials, Booking Calendar, Footer) on Cloudflare free tier. Dynamic content via D1/R2, inline editing, Google Calendar scheduling with Meet links, Resend emails, Turnstile anti-bot, Cloudflare Access admin auth, Google Drive materials sharing.

Workspace greenfield (only .git/.gitignore) — full scaffold needed.

Deployment: **GitHub repo → Cloudflare Pages automatic integration** (default GitHub connector). No manual `wrangler pages publish` in CI — GitHub push triggers Pages build.

## GitHub + Cloudflare Deployment Model

### Why GitHub + Cloudflare Pages Integration
Cloudflare Pages has first-class GitHub connector — you authorized it once in Cloudflare dashboard. After that:
- **No manual `wrangler pages publish`** needed in CI/CD
- **Push to GitHub** → Cloudflare webhook → auto-build
- **PR open/sync** → auto **Preview Deployment** with unique URL `https://<commit-hash>.portfolio-site.pages.dev` + GitHub check status + comment on PR
- **Merge to main** → auto **Production Deployment** → `https://yoursite.com` (custom domain)
- `wrangler` still used **locally only** for `pages dev` + D1/R2 local + migrations + secrets

### One-time GitHub ↔ Cloudflare Setup with Alpha (Slice 0, documented in README)

**A. GitHub repo prep (5 min):**
1. Create repo `portfolio-site` on GitHub (or use existing FanWebApp repo renamed) — default branch `main`
2. First commit will be scaffold from Slice 0
3. Create branch `alpha` from `main`: `git checkout -b alpha && git push -u origin alpha`
4. GitHub → Repo → Settings → Branches → Add rules:
   - `main`: Require PR, Require status checks (CI test + Cloudflare Pages), Require up-to-date, No direct push (optional)
   - `alpha`: Require PR for slices? Or allow direct? Recommended: Require PR but allow alpha deploys to auto publish without extra approvals (so preview PR → alpha auto). Set `alpha` protection lighter: allow force push false, require CI but not Pages? Your choice, we doc both. Simplest: protect both, but alpha allows faster merge.
5. No need to manually install Cloudflare GitHub App — done in next step when you connect Pages to GitHub (it OAuth installs app to repo).

**B. Cloudflare Pages Project Create + Connect to GitHub (10 min):**
1. Cloudflare Dashboard → Workers & Pages → Create application → Pages → Connect to Git → Select GitHub account → authorize → Select repo `portfolio-site`
2. Configure build:
   - Project name: `portfolio-site`
   - Production branch: `main`
   - Framework: Vite
   - Build command: `npm run build`
   - Build output dir: `dist`
   - Root dir: `/`
   - Build env var: `NODE_VERSION=20`, `NPM_VERSION=10`
3. Click Save & Deploy (first deploy will fail until we push code, that's ok)
4. Settings → Builds & deployments:
   - Preview deployments: Enable for all branches (default) — this gives `<hash>.portfolio-site.pages.dev` per PR
   - Branch deploy controls: Allow deployments for `main` and `alpha` and PR branches
   - Build watch paths: include all
5. Settings → Functions → Compatibility date: `2024-01-01` (in wrangler.toml also)
6. Settings → Environment Variables — THIS is where you configure 3 envs (see D1/R2 isolation section below). But initial for Slice 0, only need DB + R2 bindings, no Google APIs yet.

**C. Bindings — 3 envs (Preview, Alpha, Prod) — Dashboard UI:**

Cloudflare Dashboard → Pages → `portfolio-site` → Settings → Functions → Bindings →

*For Production (main → somewebsite.com):*
- Add D1 binding: Variable `DB`, D1 database `portfolio-db` (create first via wrangler, or create via dashboard D1 → Create)
- Add R2 binding: Variable `R2_BUCKET`, bucket `portfolio-images`
- Add Vars: 
  - `ENVIRONMENT=production`, `SITE_URL=https://somewebsite.com`, `BOOKING_CALENDAR_ID=<prod-id>`, `PERSONAL_CALENDAR_ID=<personal>`, `WORKING_HOURS_START=09:00`, `WORKING_HOURS_END=17:00`, `WORKING_DAYS=1,2,3,4,5`, `SLOT_DURATION_MINUTES=30`
  - `TURNSTILE_SITE_KEY` (public, Turnstile widget site key)
- Add Encrypted vars (Secrets) → later slices: `GCAL_SERVICE_ACCOUNT_KEY`, `RESEND_API_KEY`, `TURNSTILE_SECRET_KEY`

*For Preview (PRs → <hash>.pages.dev):*
- Same UI but switch Environment tab to Preview:
  - D1 `portfolio-db-preview`, R2 `portfolio-images-preview`
  - Vars: `ENVIRONMENT=preview`, `SITE_URL=preview` (Pages overrides with actual preview URL at runtime? set placeholder, code uses request URL fallback), `BOOKING_CALENDAR_ID=<preview-calendar-id>` (could be alpha calendar reused or separate preview calendar)
  - Secrets: same keys but you can set preview-specific values if desired

*For Alpha — custom domain branch:*
Option 1 (recommended if Pages dashboard now supports branch-specific vars): Settings → Custom domains → branch deploy vars — if you see branch selector, select `alpha` branch, then set D1 `portfolio-db-alpha`, R2 `portfolio-images-alpha`, vars `ENVIRONMENT=alpha`, `SITE_URL=https://alpha.somewebsite.com`

Option 2 (wrangler.toml envs): Define `[env.alpha]` in wrangler.toml as shown in D1/R2 isolation section, then ensure Pages builds alpha with `env=alpha`. Some Pages setups respect `env` var; we will verify in Slice 0 and document. Fallback: treat alpha.somewebsite.com as using Preview bindings logically but point to alpha DB by setting Preview env's DB to alpha DB after initial testing — then split later. We will note actual UI found in Slice 0 README.

If dashboard has no alpha branch var UI yet, you can also set via `wrangler secret put ... --env alpha` and Pages will have alpha env secrets.

**D. Custom Domains — prod + alpha (2 min):**
1. Pages → `portfolio-site` → Custom domains → Set up custom domain:
   - Prod: Enter `somewebsite.com` → select branch `main` → Continue → Cloudflare auto adds DNS (if domain on Cloudflare) CNAME + provisions SSL → green
   - Also add `www.somewebsite.com` → redirect to apex or alternate, choose
2. Add second custom domain: Enter `alpha.somewebsite.com` → select branch `alpha` (dropdown shows available branches after you pushed alpha branch) → Continue → DNS auto → SSL → green
   - Now pushing to `alpha` branch auto deploys to `https://alpha.somewebsite.com`
   - Cloudflare will show: `somewebsite.com` → production (`main`), `alpha.somewebsite.com` → branch `alpha`, `*.portfolio-site.pages.dev` → preview branches

- Each has isolated D1/R2
- Push flow fully automatic via GitHub
- You test on preview URL for dev, alpha domain for QA, prod for real.

### Git Branching + Environments (GitHub + Alpha)

**Environments:**

| Branch | Cloudflare Mapping | Custom Domain | D1 | R2 | Purpose |
|--------|-------------------|---------------|-----|-----|---------|
| `slice/*` or any feature branch | Preview Deployment | `https://<hash>.portfolio-site.pages.dev` auto | `portfolio-db-preview` | `portfolio-images-preview` | Dev testing per PR, isolated |
| `alpha` | Branch Deployment (configured as separate env) | `https://alpha.somewebsite.com` | `portfolio-db-alpha` | `portfolio-images-alpha` | QA / stakeholder testing before prod |
| `main` | Production Deployment | `https://somewebsite.com` | `portfolio-db` | `portfolio-images` | Production |

**Yes, alpha.somewebsite.com is possible:** Cloudflare Pages lets you assign a custom domain to a specific branch (not just production). Steps:
- Pages → project → Custom domains → Add custom domain → `alpha.somewebsite.com` → select branch `alpha` → Cloudflare auto adds DNS CNAME and SSL
- Pages → Settings → Branch deployments → enable for `alpha` + all preview branches
- Bindings: Set Production env for `main`, and also add env overrides for `alpha` branch (or use Preview env that applies to all non-main, but best to create separate alpha DB/R2 for isolation)

**New recommended GitHub flow with alpha:**

```
slice/0-infra-proof ──┐
slice/1-content...  ──┼─→ PR → alpha → preview https://<hash>.pages.dev (dev check)
                      ↓
                    alpha branch → auto deploy to alpha.somewebsite.com (QA/stakeholder) 
                      ↓ PR alpha → main (when green)
                    main → auto deploy to somewebsite.com (prod)
```

**Detailed per slice steps (GitHub flow with alpha):**
1. `git checkout -b slice/N-name` from `alpha` (or main then PR to alpha — your choice, we document both)
2. TDD red→green locally: `npm test`, `wrangler pages dev --d1=DB_ALPHA --r2=R2_ALPHA --local` for alpha-like local, plus preview
3. `git push origin slice/N-name`
4. GitHub → PR open vs `alpha` (not directly main, so alpha acts as integration)
5. GitHub Actions CI runs `npm test` → must pass
6. Cloudflare Pages Git integration detects PR → auto builds → posts Preview URL as GitHub check + PR comment: `Preview: https://<hash>.portfolio-site.pages.dev` (uses preview DB/R2)
7. You test on preview URL (fast dev verification)
8. Review → merge PR into `alpha` → GitHub push to `alpha` → Pages auto deploys `alpha` branch → `https://alpha.somewebsite.com` updates (this is your alpha env testing — stakeholders, full integration with alpha D1/R2 and secrets)
9. Test on `alpha.somewebsite.com` — real domain, real alpha DB, alpha secrets, full smoke e2e
10. If alpha looks good → PR `alpha` → `main` (or merge alpha to main via GitHub) → Pages auto production deploy → `https://somewebsite.com` updates
11. Smoke prod `https://somewebsite.com/api/health`
12. Rollback: revert in GitHub or redeploy previous in Pages → auto

**Alternative simpler flow if you want fewer merges:**
- Feature PR → main directly for preview, but also keep `alpha` branch that you manually sync from main (`git merge main into alpha` or GitHub sync). Then alpha.somewebsite.com always mirrors main + extra testing before you announce prod. But recommended flow above (feature → alpha → main) is safer.

**D1/R2 Isolation — 3 envs, not 2 (full isolation):**

| Purpose | Branch | D1 ID | R2 bucket | Why separate? |
|---------|--------|-------|-----------|---------------|
| Preview (PR) | `slice/*`, any PR | `portfolio-db-preview` | `portfolio-images-preview` | PRs can break data, safe to reset |
| Alpha | `alpha` | `portfolio-db-alpha` | `portfolio-images-alpha` | QA data, stakeholder testing, seed realistic data |
| Prod | `main` | `portfolio-db` | `portfolio-images` | Real bookings/contacts |

Could use 2 envs (preview+prod where alpha reuses preview), but 3 is recommended for your flow — gives you clean alpha that persists across PRs unlike ephemeral PR previews.
